# Vizual-X AutoBuild Packaging, Connector, and Environment Specification

## Purpose

Define the final package output generated by the Vizual-X AutoBuild workflow after intake, discovery, benchmarking, builder-doc generation, and Vercel runtime planning are complete.

## Final Package Objective

The workflow should produce a system package that can be:

- stored inside Vizual-X;
- inspected by the operator;
- reused as a build blueprint;
- handed to a developer or implementation agent;
- used to stage a sandbox build;
- used to prepare production requirements later.

## Package Types

The form supports:

```text
repo_package
client_spec_package
deployment_package
all
```

## Package Manifest

Every final package should include:

1. package id;
2. project name;
3. operator/client name;
4. generated timestamp;
5. intake summary;
6. benchmark summary;
7. builder docs index;
8. workflow stage summary;
9. Vercel runtime plan;
10. cron plan;
11. workflow orchestration plan;
12. sandbox build plan;
13. agent map;
14. connector map;
15. environment variable table;
16. database/state table;
17. outstanding blockers;
18. next implementation actions.

## Connector Specification

The workflow should create a connector requirements table:

| Field | Description |
|---|---|
| connector_name | service/platform name |
| purpose | why the system needs it |
| auth_type | OAuth, API key, webhook, none |
| required | yes/no |
| current_status | available, missing, future, not_required |
| data_read | what is read |
| data_write | what is written |
| risk_level | low, medium, high |
| implementation_notes | how to wire it |

## Common Connector Categories

Generated packages may include, depending on intake:

- GitHub;
- Vercel;
- Supabase/Postgres;
- Google Drive;
- Shopify;
- Gmail;
- Google Calendar;
- Slack;
- social media tools;
- video generation tools;
- financial/public data APIs;
- custom webhooks.

## Environment Variable Specification

The workflow should create an environment table:

| Variable | Purpose | Required | Scope | Sensitive | Source |
|---|---|---|---|---|---|
| example | example description | yes/no | preview/production/both | yes/no | provider |

## Baseline Vercel / Vizual-X Env Categories

Depending on implementation, the package may specify:

- database URL;
- Vercel AI Gateway key or server-side gateway auth pattern;
- connector credentials;
- public frontend flags;
- webhook secrets;
- internal cron keys where used;
- external API keys where approved.

## Important Rule

The AutoBuild workflow does **not** invent actual secret values. It creates the specifications for what must exist.

## Database and State Specification

The generated package should identify:

- data entities;
- required tables;
- state transitions;
- job/execution tables;
- document package tables;
- workflow result tables;
- connector status tables where needed.

## Recommended AutoBuild Package State Records

- intake record;
- discovery plan;
- benchmark result;
- builder-doc package;
- workflow execution state;
- cron spec;
- sandbox spec;
- agent spec;
- connector spec;
- environment spec;
- package manifest.

## Sandbox Build Output

The package must state:

- what is ready to build in sandbox;
- what remains specification-only;
- what connector/env gaps block implementation;
- what tests are required;
- what outputs are stored for later implementation.

## Final Package UI in Vizual-X

The workflow result page should show:

- package completion score;
- documents generated;
- connectors required;
- env vars required;
- sandbox build readiness;
- blockers;
- next implementation actions;
- package archive status.

## Save-in-System Requirement

The resulting package must be stored as a durable system record rather than treated as a one-time chat result.

Recommended durable entities:

- `autobuild_intakes`
- `autobuild_runs`
- `autobuild_documents`
- `autobuild_connector_specs`
- `autobuild_env_specs`
- `autobuild_packages`

## Package Completion Criteria

The final package is complete only when:

1. intake record exists;
2. discovery/benchmark output exists;
3. builder docs package exists;
4. Vercel runtime specs exist;
5. workflow/sandbox/agents/cron specs exist;
6. connector spec exists;
7. environment spec exists;
8. package manifest exists;
9. workflow state is saved;
10. Vizual-X can display the package outcome.
